# Исполнительное резюме  
**Цель:** создать в Obsidian плагин «мониторинг-сервис» для руководителя отдела, который автоматически интегрируется с корпоративным LLM и Outlook (Microsoft Graph), анализирует входящие письма, создает заметки по шаблонам и предоставляет понятный UI с кнопками. Успех измеряется работоспособностью: автоматическая синхронизация почты → заметки, генерация сводок/отчетов через LLM, простота интерфейса и соблюдение требований безопасности.  

**Ключевые результаты:**  
- Оболочка плагина с конфигом (manifest.json) и разграниченными правами.  
- Обработка почты через Интеграцию win32com и создание Markdown-заметок по шаблонам.  
- Вызовы корпоративного LLM через API (формат OpenAI-сompatible, модель) для анализа и суммаризации.  
- Удобные кнопки в Obsidian (ribbon, командная палитра) с понятными иконками и подсказками.  
- Полный чеклист тестирования (юнит, интеграция, e2e, безопасность) и инструкция по развертыванию.  

# Высокоуровневая цель и критерии успеха  
- **Главная цель:** плагин как единая точка управления: он периодически или по нажатию на кнопку читает Outlook-почту (Graph API), обрабатывает письма через LLM, а результаты аккуратно сохраняет в заметках Obsidian по шаблонам.  
- **Критерии успеха:**  
  - Письма из выбранной почтовой папки надёжно импортируются в формате заметок.  
  - Генерация отчётов/сводок LLM работает корректно (например, «Подытожь этот кейс»).  
  - UI-интерфейс — кнопки и команды работают, понятны и соответствуют нейминг-конвенциям.  
  - Все сетевые запросы (к LLM и Graph) проходят по HTTPS, а токены/API-ключи хранятся безопасно в настройках Obsidian.  
  - Плагин соответствует требованиям GDPR/IT-политики (нет утечек, логи безопасны, все вызовы проходят через корпоративные сервисы).

# Входные данные, параметры конфигурации и примеры API-вызовов  
- **Обязательные входы от пользователя:**  
  - API-ключ и базовый URL LLM (например, `https://api.corp.ai/v1`).  
  - Настройки: папка Outlook (Inbox или публичная папка), период опроса (в минутах).  

- **Параметры LLM-API:**  
  - `baseUrl` – базовый URL API (конфигурация).  
  - `model` – имя модели (напр. `"company-llm"`).  
  - `apiKey` – заголовок `Authorization: Bearer <API_KEY>`.  
  - Пример запроса к LLM (используя формат ChatGPT):  
    ```json
    POST {LLM_BASE_URL}/v1/chat/completions
    {
      "model": "company-llm",
      "messages": [
        {"role": "user", "content": "Summarize the following incident: ..."}
      ],
      "max_tokens": 200,
      "temperature": 0.7
    }
    ```  


# Манифест плагина (manifest.json) и запрашиваемые возможности  
Манифест должен содержать основные поля:  
- `id`: уникальный идентификатор плагина (например, `"OMIS-plugin"`).  
- `name`, `author`, `description`, `version` (SemVer).  
- `minAppVersion`: минимум версии Obsidian (например, `"0.14.0"`).  
- `isDesktopOnly`: `true`, т.к. нужен доступ к Node/Electron API.  
- `authorUrl`, `fundingUrl`: при желании ссылки на сайт автора/донаты.  

Дополнительно указываем:  
- **Команды (commands):** список команд Command Palette: `import-mail`, `summarize-note`, `generate-report` и т.д. Каждая команда получает `id` и `name`. (В самом коде регистрируются через `this.addCommand`).  
- **Кнопки (ribbon icons):** добавляются программно (пример см. [37†L3-L9]). В манифесте нет отдельного поля, но в описании надо упомянуть, что плагин использует `addRibbonIcon()` для своих кнопок.  
- **Настройки (settings):** опиши конфигурационные поля, которые плагин регистрирует в Settings Tab Obsidian:  
  - `mailFolder` (строка) – имя папки почты, например `"Inbox"`.  
  - `pollInterval` (число) – интервал опроса Outlook в минутах.  
  - `llmApiKey` (строка), `llmBaseUrl` (строка), `llmModel` (строка) – для LLM.  

Пример базового содержимого manifest.json:  
```json
{
  "id": "monitoring-plugin",
  "name": "Monitoring Plugin",
  "version": "1.0.0",
  "author": "ВашеИмя",
  "description": "Integrate Outlook mail and corporate LLM for automated incident notes",
  "minAppVersion": "0.14.0",
  "isDesktopOnly": true
}
```  
В описании укажем, что плагин добавляет пункты в меню, кнопки и использует разрешения, перечисленные выше.

# Интеграция с LLM API  
- **Базовый URL и модель:** URL (например, `https://api.corp-ai.local/llm`) и имя модели (напр. `company-llm-chat`) задаются в настройках.  
- **Формат запросов:** Предполагаем совместимость с API OpenAI: отправляем `POST /v1/chat/completions`.  
- **Параметры:** `model`, `messages`, `max_tokens`, `temperature`.  
- **Авторизация:** `Authorization: Bearer ${LLM_API_KEY}`.  
- **Пример JSON для суммаризации:**  
  ```json
  {
    "model": "company-llm-chat",
    "messages": [
      {"role": "user", "content": "Составь краткий отчет по этим авариям за неделю:\n- Авария1: описание...\n- Авария2: описание..."}
    ],
    "max_tokens": 150
  }
  ```  
- **Rate limits:** неизвестны, но обработка 429: при получении `error.code = "TooManyRequests"` нужно ждать и ретраить (с экспоненциальной задержкой).  
- **Шаблоны запросов (Prompt templates):**  
  - *Суммаризация инцидента:* `"Дай краткий отчет по следующему инциденту для руководства: <содержимое инцидента>."`  
  - *Анализ проблем:* `"Проанализируй причины следующих повторяющихся сбоев сети:"` + список.  
  - *Формирование задачи:* `"Сформируй план задач по результатам инцидента: <инфо>."`  

# UX-спецификация для кнопок и интерфейса  
- **Ribbon Icons:** на боковой панели Obsidian (ribbon) добавляются кнопки:  
  - 📧 **«Import Mail»** – иконка конверта. При клике запускает чтение почты из Graph и создание заметок из новых писем【37†L3-L9】. Tooltip: «Импортировать почту».  
  - 📈 **«Weekly Report»** – иконка графика или отчёта. Генерирует сводный отчёт по активности (с помощью LLM) и открывает Markdown-заметку с отчётом. Tooltip: «Сформировать еженедельный отчет».  
  - 📝 **«Summarize Note»** – иконка текста. В контексте открытой заметки запрашивает у LLM краткое резюме её содержимого. Tooltip: «Сделать сводку заметки»【37†L3-L9】.  
- **Командная палитра:** регистрируем команды-псевдонимы (`id` и `name`) для тех же действий, чтобы их можно было вызывать через Ctrl+P. Пример:  
  ```ts
  this.addCommand({
    id: 'import-mail',
    name: 'Обновить почту (Import Mail)',
    callback: () => this.importMail()
  });
  ```  
- **Положение кнопок:** слева в панели (ribbon), порядок можно менять. Дополнительно, можно вывести кнопку в titleBar (верх заметки) или status bar для быстрого доступа к текущей функции (например, «Summarize current note»).  
- **Внешний вид:** используем встроенные иконки lucide или SVG. Например, `this.addRibbonIcon('mail', ...)`. Стилистически они должны сочетаться с темой Obsidian (белые/цветные иконки в зависимости от темы).  
- **Доступность:** каждому `addRibbonIcon` передаётся `name` – он автоматически используется как aria-label для screen-reader’ов【37†L3-L9】. Используем лаконичные описания на русском («Импорт почты», «Отчет»).

# Чеклист тестирования  
- **Unit-тесты:**  
  - Парсинг JSON-писем → объекты. Пример: смоделировать ответ outlook и проверить правильное извлечение subject/from/body.  
  - Функции создания заметок: на входе данные письма, ожидаемый Markdown (по шаблону).  
  - LLM-интеграция: эмулировать ответ LLM и убедиться, что результат корректно вставляется в заметку.  
- **Integration-тесты:**  
  - Модуль `importMail()`: подменить письмо на мок, проверить создание файлов в тестовом плагин-хранилище.  
  - Модуль `generateReport()`: подменить LLM API на статичный ответ, проверить итоговую структуру отчёта.  
  - Обмен данными: (OBSIDIAN) при нажатии кнопки вызывается нужный метод; (Settings) проверка, что ключи сохраняются.  
- **E2E-тесты:**  
  - Окружение: тестовый vault. Симуляция потока: нотификация об апдейте почты → запуск импорта → открытие созданной заметки с верным контентом.  
  - Сценарий «Summarize Note»: пользователь выделил текст, нажал кнопку, появилось модальное окно с результатом (или вставлен текст).  
- **Security-тесты:**  
  - Убедиться, что запросы идут по HTTPS.  
  - Нет логирования чувствительных данных (токен, письма) – проверить логи/консоль.  
  - Simulate манипуляции: например, попытка инъекции через содержимое письма – убедиться в экранировании.  
- **Performance:**  
  - Замер времени обработки большого письма.  
  - Нагрузочное: симулировать множественные письма (10+, 50+) – проверить реакцию (устойчивость UI, ошибки).

- **Архив и бэкап:** Храним созданные заметки в поддиректории (напр. `Vault/OMIS`), можно ручной перенос старых заметок в архив (`/Archive`).  
- **Assumptions (предположения):**
